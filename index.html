For Every Path

The non-negotiables

Regardless of experience — skip any of these and you are building on sand.

  • 01

    Write a PRD First

    Vague input produces vague code. Define the problem, user, features, and success metrics before opening any AI tool.

    • Problem statement
    • Target user persona
    • Must-have features (max 5)
    • User flows
    • Success metrics
  • 02

    Git from Day One

    Initialize a repo and push to GitHub before writing a single prompt. AI will break things — you must be able to roll back.

    • Initialize Git immediately
    • Push to GitHub
    • Commit after every feature
    • Never deploy manually
  • 03

    Security Review

    45% of AI-generated code contains OWASP Top 10 vulnerabilities. Review auth, inputs, SQL, and secrets after every major feature.

    • Auth on all protected routes
    • Validate all user input
    • No hardcoded secrets
    • Rate limiting on public APIs
  • 04

    Build Thin Verticals

    One complete end-to-end feature, tested fully, before the next. Smaller context = better AI output.

    • One feature per session
    • Test before moving on
    • End-to-end first
    • Then polish
  • 05

    Validate Before Building

    Find real people complaining about your problem on Reddit, X, and G2 before writing a single line. Those complaints are your roadmap.

    • Reddit threads
    • G2 / Trustpilot reviews
    • Competitor complaints
    • Talk to 5 real people
  • 06

    Add Observability Early

    Add error tracking and analytics before users arrive — not after they complain about something that has been broken for weeks.

    • Sentry (errors)
    • PostHog (analytics)
    • Logtail (logs)
    • Uptime monitoring
  • 07

    Scope Aggressively

    5 features done perfectly beats 50 features poorly. Define your MLP scope before you start. Ship, learn, then iterate.

    • Define MLP scope first
    • Defer nice-to-haves
    • No AI features at v1
    • Ship → Learn → Iterate
  • 08

    5 Users Before Features

    Talk to 5 real users before adding anything new. Where they get confused is your actual roadmap — not your wishlist.

    • Observe, do not explain
    • Note friction points
    • Fix confusion first
    • Then add features
